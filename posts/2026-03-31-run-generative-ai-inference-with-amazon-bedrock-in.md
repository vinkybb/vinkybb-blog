# Amazon Bedrock 在新西兰区域上线：亚太地区生成式 AI 推理的新纪元

## 背景介绍

生成式 AI 正在重塑各行各业，而模型推理的区域部署成为企业关注的核心议题。长期以来，新西兰地区的 AWS 客户面临一个现实困境：要使用 Amazon Bedrock 的基础模型服务，必须调用其他区域的 API，这不仅带来延迟问题，更触及数据合规的敏感神经。

2026 年 3 月 26 日，这个困境迎来终结。亚马逊正式宣布 **Amazon Bedrock 服务在新西兰亚太区域（ap-southeast-6）正式可用**。新西兰本地客户现在可以直接从奥克兰区域访问 Anthropic Claude 系列模型和 Amazon Nova 2 Lite 模型，无需跨境数据传输。这意味着更低的延迟、更高的合规性，以及更流畅的开发体验。

## 核心内容：可用模型与跨区域推理机制

### 可用模型清单

新西兰区域现已支持两类主流模型家族。**Anthropic Claude 系列**涵盖了完整的产品矩阵：Claude Opus 4.5 和 4.6（高端推理）、Claude Sonnet 4.5 和 4.6（平衡性能）、以及 Claude Haiku 4.5（轻量高效）。同时，**Amazon Nova 2 Lite** 也已上线，为成本敏感型应用提供本土选项。

值得注意的是，所有模型均支持 **跨区域推理** 功能。这项能力允许 Bedrock 将推理请求智能分发到多个 AWS 区域，从而在流量高峰期实现更高吞吐量，避免单一区域容量瓶颈导致的请求拒绝。

### 跨区域推理的工作原理

跨区域推理是 Amazon Bedrock 的核心能力，它将推理处理分布到多个 AWS 区域以提升规模吞吐。当开发者调用一个跨区域推理 Profile 时，Bedrock 会将请求从源区域（API 发起地）路由到目标区域（实际推理执行地）。

关键安全保障贯穿整个流程：所有跨区域传输数据均通过 AWS 全球专有网络，不经过公共互联网；传输过程全程加密；所有请求日志记录在源区域的 AWS CloudTrail 中；如配置模型调用日志，数据将发布到源区域的 CloudWatch Logs 或 Amazon S3。

Bedrock 提供两类跨区域推理 Profile：**地理路由**和**全球路由**。

地理路由将请求限制在特定地理边界内。以 AU Profile 为例，从奥克兰发起的请求可路由至奥克兰、悉尼或墨尔本三个区域，推理处理全程留在澳新地理范围内。这专为有数据驻留要求的组织设计，确保合规性。

全球路由则将请求分发到所有支持的 AWS 商业区域，提供最高吞吐能力。这适合没有严格数据驻留限制的组织，能最大化利用全球推理容量。

## 技术分析：新西兰作为源区域的创新架构

### AU 地理路由配置详解

随着新西兰区域的上线，AU 跨区域 Profile 现已覆盖澳新三区域。奥克兰（ap-southeast-6）作为源区域，请求可路由至三个目标区域：ap-southeast-6（奥克兰本地）、ap-southeast-2（悉尼）、ap-southeast-4（墨尔本）。这为新西兰客户提供了三倍容量弹性，而悉尼和墨尔本的 Profile 配置保持不变，继续在二者之间路由。

一个重要细节：现有的澳大利亚源区域配置不受影响，新西兰的加入不会改变悉尼和墨尔本的目标区域范围。这体现了向后兼容的设计理念。

### 全球路由的优势

从奥克兰发起全球跨区域推理，提供两大核心价值：一是更高吞吐量，智能路由动态将流量分发到全球支持区域，显著降低流量峰值时的 throttling 概率；二是内置韧性，请求自动路由到有可用容量的区域，帮助应用在需求变化时保持连续运行。

### 安全与合规考量

跨区域推理设计以安全为核心。所有请求通过 AWS 全球网络传输，端到端加密；数据静态存储保持在源区域。对于使用服务控制策略（SCP）限制区域访问的组织，需注意：AU 地理路由要求 SCP 允许 ap-southeast-2、ap-southeast-4、ap-southeast-6 三个区域的 Bedrock 操作；全球路由则需额外允许 ` unspecified` 作为区域值，因为目标区域动态确定。

Amazon Bedrock 在新西兰区域的上线，标志着亚太地区生成式 AI 基础设施的重大完善。本地区域部署带来三重价值：降低推理延迟、满足数据驻留合规要求、提供熟悉的 Claude 和 Nova 模型选择。跨区域推理机制进一步增强了弹性与吞吐能力，而地理路由选项为合规敏感型组织提供了精准控制。
