# Systematic debugging for AI agents: Introducing the AgentRx framework

# AgentRx：AI Agent 系统化调试框架

## 背景：AI Agent 调试的困境

随着 AI agent 在自动化任务中的广泛应用，其调试问题日益凸显。一个典型的 agent 执行轨迹可能包含数十个步骤，涉及多个 agent 之间的交互协作，且具有强烈的随机性——同样的任务在不同执行中可能产生截然不同的结果。当 agent 失败时，传统的"任务是否完成"的二元评判指标无法揭示失败的确切位置和根本原因，开发者往往陷入漫长的排查过程，在海量日志中艰难定位真正的根因。

这种调试困境源于三个核心特征：**执行轨迹长周期**、**行为随机性强**、**多 agent 交互复杂**。一条失败轨迹可能包含 30-50 个步骤，每个步骤都可能产生分支，而真正的错误可能隐藏在第 42 步的一个工具调用参数中，被数十个看似正常的操作所掩盖。传统调试方法依赖人工逐步检查，既耗时又容易遗漏关键信息。

## AgentRx 框架：证据驱动的诊断方法论

微软研究院开源的 **AgentRx** 框架，提供了一套自动化、领域无关的系统化调试方案。其核心理念是从"猜测错误"转向"证据驱动诊断"，通过四阶段流水线精准定位"**关键失败步骤**"——首个不可恢复的错误发生点。

### 第一阶段：轨迹标准化

异构的 agent 框架产生的日志格式各不相同，有的使用 JSON，有的采用纯文本，有的混合多种格式。AgentRx 首先将这些异构日志转换为统一的中间表示，建立标准化的执行轨迹模型。这一步骤确保后续分析可以在统一的数据结构上进行，不受原始日志格式差异的影响。

### 第二阶段：约束合成

基于工具的 schema 定义和领域策略，AgentRx 自动生成可执行的验证约束。例如，当 agent 调用一个返回 JSON 数据的 API 时，系统自动合成"响应必须是有效 JSON"的约束；当工具要求特定参数格式时，生成相应的参数验证规则。这些约束不是硬编码的，而是根据工具的接口定义动态生成，确保验证逻辑与实际工具行为保持一致。

### 第三阶段：守护式评估

系统沿着执行轨迹逐步验证合成约束，生成详细的审计日志。当某个步骤违反约束时，审计日志记录完整的违规证据——包括违反的具体约束、违规时的上下文状态、以及导致的后果。这种"守护"式的实时评估，将潜在问题立即捕获，而非事后追溯。

### 第四阶段：LLM 仲裁

最后阶段结合验证日志和预定义的失败分类体系，利用大语言模型识别关键失败步骤。系统将审计证据、失败类别（如规划偏差、幻觉输出、工具调用错误、上下文丢失等）和轨迹上下文提供给 LLM，使其能够综合分析并给出根因判定。这一步骤的关键在于：LLM 不是凭空判断，而是基于前三阶段积累的硬证据进行裁决。

## 基准测试与性能验证

AgentRx 团队构建了专门的基准数据集——**AgentRx Benchmark**，包含 115 条人工标注的失败轨迹，涵盖三个主流 agent 系统：τ-bench、Flash 和 Magentic-One。这些轨迹被标注为九大失败类别：规划偏差、幻觉输出、工具调用错误、上下文丢失、执行顺序混乱、参数验证失败、环境状态不一致、多 agent 协调失败、以及资源约束违反。

在该基准上的评测结果显示，AgentRx 相比基于提示的传统方法，**关键失败步骤定位准确率提升 23.6%**，**根因归因准确率提升 22.9%**。这一性能增益在复杂多步骤场景中尤为显著——当轨迹长度超过 30 步时，AgentRx 的优势进一步扩大，证明其能够有效处理传统方法难以应对的长轨迹调试问题。

## 技术分析：从方法论到工程实践

AgentRx 的设计体现了系统化调试的三个关键原则：

**结构化验证优于事后猜测**。传统方法往往在失败发生后才开始回溯排查，而 AgentRx 通过守护式评估实时捕获违规证据，将调试前置化。这种设计类似于软件工程中的"持续集成测试"理念——问题越早发现，定位成本越低。

**领域无关的通用性**。框架不依赖特定 agent 框架或工具集，任何符合基本接口约定的系统都可以接入。这种通用性来自约束合成阶段的动态生成机制——验证逻辑从工具 schema 自动推导，而非人工预设。

**可审计的调试链路**。从轨迹标准化到 LLM 仲裁，每个阶段都产生可追溯的中间结果。开发者可以审查审计日志、验证约束的合理性、检查 LLM 的推理过程。这种透明性对于构建可信的智能系统至关重要——调试过程本身必须可解释、可审计。

## 总结与展望

AgentRx 为 AI agent 系统工程提供了从"直觉调试"到"证据驱动诊断"的范式转换。其四阶段流水线设计将原本依赖经验的排查过程转化为结构化的分析流程，显著提升了失败定位和根因识别的准确性。框架已开源（代码与数据可通过 https://aka.ms/AgentRx/Code 获取），为构建可审计、可信赖的智能系统提供了实用的工程工具。

未来，随着 agent 系统在复杂场景中的进一步应用，系统化调试将成为不可或缺的基础设施。AgentRx 的方法论——轨迹标准化、约束合成、守护评估、智能仲裁——为这一方向建立了可复用的技术范式，为后续的调试工具研发提供了参考架构。从"猜测错误"到"证据驱动"，AI agent 的调试能力正在走向成熟。

# AgentRx：AI Agent 系统化调试框架

## 背景：智能系统调试的透明性困境

当人类犯错时，我们通常可以追溯其逻辑链条。但当 AI agent 在五十步任务的第十步偏离安全策略或幻觉式地捏造工具输出时，准确定位问题发生的时刻和原因却是一项艰巨的工程挑战。随着 AI agent 从简单聊天机器人进化为能够管理云事件、导航复杂 Web 界面、执行多步 API 工作流的自治系统，调试已成为制约其可靠部署的核心瓶颈。

现代 AI agent 具有三个显著特征：**长周期性**——在扩展时段内执行数十个动作；**概率性**——相同输入可能产生不同输出，复现困难；**多 agent 协作性**——失败可能在不同 agent 之间传递，掩盖原始根因。传统的成功指标（如"任务是否完成"）无法揭示失败的精确时刻。构建安全的 agent 系统需要识别轨迹变得不可恢复的确切节点，并捕获该步骤的错误证据。

## AgentRx 框架：证据驱动的诊断流水线

微软研究院开源的 **AgentRx**（Agent Diagnosis）框架将 agent 执行视为需要验证的系统追踪，采用结构化多阶段流水线替代单一 LLM 的猜测式诊断。

### 阶段一：轨迹标准化

不同领域的异构日志被转换为统一的中间表示。无论是 JSON 格式、纯文本还是混合格式的执行记录，都映射为标准化的轨迹模型，确保后续分析在一致的数据结构上进行。

### 阶段二：约束合成

框架基于工具 schema（如"API 必须返回有效 JSON 响应"）和领域策略（如"未经用户确认不得删除数据"）自动生成可执行约束。这些约束并非硬编码预设，而是从工具接口定义动态推导，保证验证逻辑与实际工具行为一致。

### 阶段三：守护式评估

AgentRx 逐步评估约束，仅在守护条件适用时检查相应约束，生成包含证据支撑违规记录的可审计验证日志。每个违规点都携带完整的上下文状态和触发条件，形成可追溯的诊断链路。

### 阶段四：LLM 仲裁

最终阶段中，LLM 仲裁器结合验证日志和失败分类体系，识别关键失败步骤——首个不可恢复错误。LLM 的判断基于前三阶段积累的硬证据，而非凭空推理。

## AgentRx Benchmark 与失败分类体系

为评估框架性能，团队构建了包含 115 条人工标注失败轨迹的基准数据集，覆盖三个复杂领域：τ-bench（结构化 API 工作流）、Flash（现实事件管理与系统排查）、Magentic-One（开放式 Web 和文件任务）。

基于扎根理论方法，研究团队推导出跨领域通用的九类别失败分类体系：

| 类别 | 描述 |
|------|------|
| Plan Adherence Failure | 忽略必要步骤或执行未规划的额外动作 |
| Invention of New Information | 改变非基于追踪或工具输出的事实（幻觉） |
| Invalid Invocation | 工具调用格式错误、缺失参数或 schema 无效 |
| Misinterpretation of Tool Output | 错误解读工具输出，基于错误假设行动 |
| Intent–Plan Misalignment | 误读用户目标或约束，规划错误 |
| Under-specified User Intent | 因缺少必要信息无法继续执行 |
| Intent Not Supported | 无可用工具可完成请求任务 |
| Guardrails Triggered | 执行被安全或访问限制阻断 |
| System Failure | 连接性或工具端点故障 |

分析显示，在 Magentic-One 等多 agent 系统中，轨迹常包含多个错误点。AgentRx 聚焦识别首个关键违规，确保根因定位的精确性。

## 性能验证与方法论价值

实验结果显示，AgentRx 相比现有 LLM 提示基准实现显著提升：**失败定位准确率绝对提升 23.6%**，**根因归因准确率提升 22.9%**。

通过可审计日志揭示失败的"原因"，AgentRx 使开发者超越试错式提示调优，转向系统化的 agent 工程实践。Agent 可靠性是现实部署的前提，框架和标注基准的开源为社区提供了构建透明、韧性智能系统的实用工具。

## 总结

AgentRx 将 agent 调试从直觉式猜测提升为证据驱动的结构化诊断。其四阶段流水线设计——轨迹标准化、约束合成、守护评估、智能仲裁——为复杂智能系统的故障定位建立了可复用的技术范式。框架代码与基准数据已开源（https://aka.ms/AgentRx/Code），支持开发者诊断自有 agent 工作流并贡献失败约束库。构建可审计、可信赖的 AI agent，需要超越能力指标的系统化调试基础设施。
