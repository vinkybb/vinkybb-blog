# 代码审查在 LLM 时代的新定位：人类对齐机制

## 背景：从绿色项目到生产环境的跨越

2025年初，大语言模型辅助编程还是一个充满争议的话题。当时普遍认为，LLM 适合全新的绿色项目，但对于生产级代码库来说，生成的代码质量"过于松散"，难以满足工程标准。然而，短短几个月后，情况发生了戏剧性的转变。

以 PyTorch 项目为例，Edward Yang 和 Jason Ansel 等核心贡献者已经大量采用 LLM 辅助提交 PR。这些合并的代码不再只是实验性的尝试，而是真正进入了生产流程。这一变化引发了业界广泛讨论：Simon Willison 提出"代码证明其有效"的观点，LLVM 社区也在重新审视 AI 政策与代码规范的边界。在这个转折点上，我们需要重新思考一个根本问题：**代码审查的本质是什么？**

## 核心观点：从纠错机制到对齐机制

传统的代码审查被视为质量把关环节，审查者的职责是发现问题、提出修改建议，作者则负责修复。但在 LLM 时代，这个逻辑需要被彻底重构。

**审查者的目标不再是单纯修复问题，而是让使用 LLM 的作者理解并认同问题的本质。**

一个典型案例是 LLM 的"过度防御性编码"倾向。模型生成的代码常常充斥着不必要的 try-catch 块，或是对变量类型的冗余检查。这些防御逻辑在局部看起来"安全"，但在系统全局视角下可能完全是多余的。如果审查者只是指出问题并要求修复，作者可能机械地接受建议，但没有真正理解为什么这些防御代码是不必要的。

真正的对齐发生在作者理解了系统约束之后。例如，某个全局不变式可能决定了变量类型始终是预期的类型，因此类型检查是多余的。当作者掌握了这个高层知识，他们就可以自行指导 LLM 修正代码，而不需要审查者逐条指出每个问题。理想状态下，审查者不需要再评论任何机械性问题，因为作者已经提前完成了对齐工作。

## 提交者的责任升级：Token 成本降低后的新标准

LLM 大幅降低了代码生成的成本，但这并不意味着作者可以降低提交标准。相反，**成本降低应当带来更高的质量预期**。

当作者使用 LLM 生成 PR 时，他们需要传递的是高层信息：代码如何工作？理解现有系统需要哪些知识？这些信息不必全部写在 PR 描述中，而是应该体现在代码本身的清晰度上。如果 LLM 提出的修复方案作者自己都难以理解，正确的做法不是强行提交，而是继续与模型迭代，直到生成的 diff "显然正确"。

一个重要的心态转变是：**不要担心"浪费"审查者的时间**。Token 是廉价的，作者可以多次重新生成代码、尝试不同的实现方案。真正投入的是理解问题的时间，而这部分工作本身就是必要的。审查者的精力应该用于更高价值的对齐讨论，而不是处理机械性问题。

## 技能权重重塑：读代码能力超越写代码能力

LLM 的崛起并非简单淘汰"不会用 AI 的工程师"，而是重新定义了工程师技能的价值权重。

传统的软件开发能力中，"机械编码能力"占据重要位置。但在 LLM 时代，这项技能的重要性大幅下降。更关键的技能变成了：**读代码能力、大局观推理能力、清晰沟通能力、以及代码品味**。

这对初级工程师的职业生涯有深远影响。存在一类"不擅长编码但擅长高层软技能"的工程师原型。在过去，他们可能面临成长瓶颈；但在新环境下，他们将变得更有价值——因为他们能理解系统、提出正确的问题、做出合理的决策，而编码执行可以交给 LLM 完成。

另一方面，审查者对"重复问题"的耐心将大幅降低。如果一个工程师在代码审查中反复出现相同的理解偏差，审查者不再会像过去那样容忍——因为"机械编码能力"本身的价值已经下降，审查者更看重的是学习和对齐的速度。

## 团队的本质价值：维护系统的共享愿景

当前的大语言模型有一个根本性的局限：**它们没有记忆**。每次调用时，模型都需要从零开始推理，重新"发现"系统的一切。

这正是人类团队的核心价值所在。团队的本质不是产出代码，而是**维护一个共享的系统愿景**——代码在柏拉图式的理想状态下应该做什么、应该如何工作。这个愿景无法存储在 LLM 中，只能存在于团队成员的共同理解里。

代码审查应当围绕这个目的重新设计。审查过程不只是检查代码是否符合规范，而是确认代码是否与团队的共享愿景对齐。当作者和审查者达成对齐，后续的机械性问题可以由作者自主处理；审查者的精力则应该用于更新和维护这个更高层的系统理解。

## 结语：从流程重构到团队进化

LLM 不是威胁，而是催化剂。它促使我们重新审视代码审查的价值，推动团队从"纠错流程"进化为"对齐机制"。在这个新范式下，审查者和作者的职责边界被重新划分：作者承担更多前期对齐工作，审查者聚焦于高层共识的建立。

这不是一场技术革命，而是一场协作模式的进化。真正的挑战不在于掌握 LLM 工具，而在于重新理解团队在技术系统中的角色定位。代码审查的未来，属于那些能够维护共享愿景的团队。
